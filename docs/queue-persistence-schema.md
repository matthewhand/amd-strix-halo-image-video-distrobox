# Queue persistence & schema

## Overview

Slopfinity persists queue items in an authoritative SQLite database (at `comfy-outputs/experiments/slopfinity.db` or `$SLOPFINITY_STATE_DIR/slopfinity.db`) with a JSON backup maintained alongside it. The queue serves as the cross-process IPC layer between the FastAPI dashboard, the fleet runner (`run_fleet.py`), and dormant worker subprocesses (Phase-4 refactor). The schema encodes both flat legacy fields and a structured per-stage status graph introduced in Phase 1 of the queue schema redesign.

## Core model: QueueItem (slopfinity/models.py)

The `QueueItem` SQLModel (lines 12–53) defines the authoritative schema:

```python
class QueueItem(SQLModel, table=True):
    id: str = Field(default_factory=lambda: str(uuid.uuid4()), primary_key=True)
    ts: float = Field(default_factory=time.time, index=True)
    schema_version: int = Field(default=1)
    status: str = Field(default="pending", index=True)
    prompt: str
    title: Optional[str] = None
    priority: str = Field(default="normal")
    concurrent: bool = Field(default=False)
    infinity: bool = Field(default=False)
    when_idle: bool = Field(default=False)
    chaos: bool = Field(default=False)
    image_only: bool = Field(default=False)
    fast_track: bool = Field(default=False)
    config_snapshot: Dict[str, Any] = Field(default_factory=dict, sa_column=Column(JSON))
    stages: Dict[str, Any] = Field(default_factory=dict, sa_column=Column(JSON))
    extra: Dict[str, Any] = Field(default_factory=dict, sa_column=Column(JSON))
    created_at: datetime = Field(default_factory=datetime.utcnow)
    completed_ts: Optional[float] = None
    cancelled_ts: Optional[float] = None
    succeeded: Optional[bool] = None
    error: Optional[str] = None
    asset_paths: List[str] = Field(default_factory=list, sa_column=Column(JSON))
    story_id: Optional[str] = None
    story_title: Optional[str] = None
```

### Key field invariants

- **id**: Stamped at write time if missing (see _split_queue_item); never churns once written. Primary key. Generated by `queue_schema.make_id()` (lines 24–25).
- **ts**: Queue insertion timestamp (float seconds). Indexed. Defaults to `time.time()`.
- **schema_version**: Defaults to 1 (legacy). Migrated to 2 by `queue_schema.migrate_legacy()`. Prevents re-migration of already-migrated items. See models.py lines 17–19 comment.
- **config_snapshot**: The configuration dict at inject time, serialized as JSON. Used by workers to resolve model defaults via the `stages` dict model assignments.
- **stages**: Phase 1 queue schema — a dict mapping stage names to `{status, model?, started_ts?, completed_ts?}`. Empty on legacy items until migrated.
- **extra**: **Catch-all JSON column** (added in commit 56d74be) that preserves fields without a dedicated column. See "Silent field drop problem" below.

## The silent field drop problem & the extra column (commit 56d74be)

### Root cause

Writers (the `/inject` endpoint and `run_fleet`) stamp fields onto task dicts that have no dedicated `QueueItem` column:
- `seed_image`, `seed_images`, `seeds_mode`, `seed_prompt` — for seed-image features
- `stage_prompts`, `stage_prompts_raw` — for stage-specific prompt overrides
- `polymorphic` — for polymorphic-model features
- `started_ts`, `requeued_from_ts` — for run_fleet's state tracking

The old `save_queue()` code used a naive filter:
```python
filtered = {k: v for k, v in item_dict.items() if k in QueueItem.model_fields}
```

Since these fields were not in `model_fields`, they were **silently dropped** before the SQLite INSERT. When `get_queue()` read from the DB first, `run_fleet` (which claims items via the DB) saw them already gone on the first roundtrip—silently breaking seed-image, FLF2V keyframe, and stage-prompt features.

### Solution: the extra catch-all column

Commit 56d74be adds a `extra: Dict[str, Any]` JSON column to `QueueItem` (models.py, line 41) with two accessor functions in `config.py`:

**_split_queue_item(item_dict)** (config.py, lines 554–574):
```python
def _split_queue_item(item_dict):
    """Split a queue dict into QueueItem(**kwargs) form, funneling any key
    without a dedicated column into the `extra` JSON catch-all.
    ...
    """
    item_dict.setdefault("id", queue_schema.make_id())
    known = QueueItem.model_fields
    extra = {k: v for k, v in item_dict.items() if k not in known}
    filtered = {k: v for k, v in item_dict.items() if k in known and k != "extra"}
    # Defensive: if an un-flattened `extra` dict slipped through, merge it
    # (newer top-level unknowns win over stale nested copies).
    prev = item_dict.get("extra")
    if isinstance(prev, dict):
        extra = {**prev, **extra}
    filtered["extra"] = extra
    return filtered
```

This function:
1. Stamps a stable id if missing (via `queue_schema.make_id()` — see id-stamping churn fix below).
2. Partitions the input dict into known columns (`filtered`) and unknowns (`extra`).
3. Merges any pre-existing `extra` dict to handle defensive re-flattening.
4. Returns the dict suitable for `QueueItem(**kwargs)`.

**_flatten_queue_item(m)** (config.py, lines 577–586):
```python
def _flatten_queue_item(m):
    """model_dump() a QueueItem and lift its `extra` catch-all back to the top
    level so callers see one flat dict (exactly as before the column existed).
    setdefault so a real column is never clobbered by a stale extra entry."""
    d = m.model_dump()
    extra = d.pop("extra", None)
    if isinstance(extra, dict):
        for k, v in extra.items():
            d.setdefault(k, v)
    return d
```

This function:
1. Dumps the SQLModel to a plain dict.
2. Extracts and removes the `extra` dict.
3. Lifts each extra key back to the top level (using `setdefault` so real columns are never clobbered).
4. Returns the flattened dict, so callers see the exact structure as before.

## Stable id stamping fix (commit 7c70d1e)

### Problem: uuid churning on every save

The QueueItem model initializes `id` with a `default_factory` (models.py, line 13):
```python
id: str = Field(default_factory=lambda: str(uuid.uuid4()), primary_key=True)
```

`run_fleet` appends rows to the queue without an explicit id. The old `save_queue()` code was:
```python
item_dict.setdefault("id", queue_schema.make_id())
```

But this line only stamped the id on the *dict* temporarily; the SQLModel's `default_factory` was invoked on *every* `QueueItem(**item_dict)` construction if the dict passed `id=None`. This caused the id to **churn (regenerate) on every save**, breaking status tracking ("working" → "done") because the same logical item changed identity across saves.

### Solution: stamp id in _split_queue_item

Commit 7c70d1e moves id stamping into `_split_queue_item()` (config.py, line 564):
```python
item_dict.setdefault("id", queue_schema.make_id())
```

Because `_split_queue_item()` is called *before* `QueueItem(**kwargs)` construction (save_queue line 634), the id is now part of the filtered dict and always passed explicitly, bypassing the `default_factory`. Status tracking now survives across saves.

**queue_schema.make_id()** (queue_schema.py, lines 24–25):
```python
def make_id() -> str:
    return f"q-{int(time.time()*1000)}-{uuid.uuid4().hex[:6]}"
```

This generates a sortable, collision-resistant id: millisecond timestamp + 6-char uuid suffix.

## Queue schema: per-stage status (queue_schema.py)

### Overview

Phase 1 introduces a **stages** dict that encodes the per-stage execution status, prerequisites, and model assignments. The legacy `status` field (a single string like "pending", "working", "done") is complemented by this graph; Phase 2+ workers will consult `stages` exclusively.

### Schema version and migration

**migrate_legacy(item)** (queue_schema.py, lines 28–69):

```python
def migrate_legacy(item: dict) -> dict:
    if item.get("schema_version") == SCHEMA_VERSION and "stages" in item:
        return item
    item.setdefault("id", make_id())
    item["schema_version"] = SCHEMA_VERSION
    legacy_status = item.get("status", "pending")
    succeeded = item.get("succeeded")
    image_only = bool(item.get("image_only", False))
    # Translate legacy status + succeeded boolean to bulk stage status.
    if legacy_status == "done" and succeeded is True:
        bulk = "done"
    elif legacy_status == "done" and succeeded is False:
        bulk = "failed"
    elif legacy_status == "cancelled":
        bulk = "skipped"
    elif legacy_status == "working":
        bulk = "needs"  # Leave as needs for coordinator pickup.
    else:
        bulk = "needs"
    # Build stages dict with model assignments from config_snapshot.
    config = item.get("config_snapshot") or {}
    stages = {}
    for s in STAGE_ORDER:
        if image_only and s in ("video", "audio", "tts", "post", "merge"):
            stages[s] = {"status": "skipped"}
        else:
            entry = {"status": bulk}
            if s == "image":
                entry["model"] = config.get("base_model", "qwen")
            elif s == "video":
                entry["model"] = config.get("video_model", "ltx-2.3")
            elif s == "audio":
                entry["model"] = config.get("audio_model", "heartmula")
            elif s == "tts":
                entry["model"] = config.get("tts_model", "qwen-tts")
            elif s == "post":
                entry["model"] = config.get("upscale_model", "ltx-spatial")
            stages[s] = entry
    item["stages"] = stages
    return item
```

Key behaviors:
- **Idempotent**: If `schema_version == SCHEMA_VERSION` and `stages` is already present, returns unchanged (line 29).
- **Stamps id if missing** (line 31).
- **Sets schema_version to 2** (line 32).
- **Derives per-stage status from legacy fields**:
  - `done + succeeded=True` → all stages "done"
  - `done + succeeded=False` → all stages "failed"
  - `cancelled` → all stages "skipped"
  - `working` → all stages "needs" (so coordinator can still claim it; legacy runner doesn't read `stages`)
  - other → all stages "needs"
- **Populates model assignments** from `config_snapshot` so workers know which model to use (lines 57–66).
- **image_only=true** skips video/audio/tts/post/merge stages (sets them to "skipped", line 54).

### Per-stage status tracking

**set_stage_status(item, stage, status, **fields)** (queue_schema.py, lines 76–83):
```python
def set_stage_status(item: dict, stage: str, status: str, **fields) -> None:
    item.setdefault("stages", {}).setdefault(stage, {})
    item["stages"][stage]["status"] = status
    item["stages"][stage].update(fields)
    if status in ("done", "failed", "skipped"):
        item["stages"][stage].setdefault("completed_ts", time.time())
    if status == "working":
        item["stages"][stage].setdefault("started_ts", time.time())
```

Automatically timestamps `completed_ts` on terminal states and `started_ts` when work begins.

### Overall status derivation

**overall_status(item)** (queue_schema.py, lines 102–113):
```python
def overall_status(item: dict) -> str:
    stages = item.get("stages") or {}
    if not stages:
        return item.get("status", "pending")  # Fallback to legacy field.
    statuses = [s.get("status", "needs") for s in stages.values()]
    if all(s in ("done", "skipped") for s in statuses):
        return "done" if any(s == "done" for s in statuses) else "skipped"
    if any(s == "failed" for s in statuses):
        return "failed"
    if any(s == "working" for s in statuses):
        return "running"
    return "pending"
```

Synthesizes a single `status` from the per-stage statuses for dashboard display.

## Dual persistence: SQLite (authoritative) + JSON (backup)

### Authoritative store: SQLite

The SQLite database (`slopfinity.db`, at `comfy-outputs/experiments/slopfinity.db` or `$SLOPFINITY_STATE_DIR/slopfinity.db`) is the primary queue source. All writers persist to it, and all readers consult it first.

- **Engine setup** (db.py, lines 14–15): SQLite with `check_same_thread=False` for FastAPI/asyncio.
- **Schema creation** (db.py, line 54): `SQLModel.metadata.create_all(engine)` on `init_db()`.
- **Column migration** (db.py, lines 17–49): Idempotent `ALTER TABLE ADD COLUMN` for new fields.

### Backup store: queue.json

Commit 56d74be documents the dual store (config.py, lines 499–525). The JSON backup is synced *after* the DB commit (config.py, lines 637–647):

```python
# Sync to JSON as backup
try:
    os.makedirs(os.path.dirname(QUEUE_FILE), exist_ok=True)
    tmp = QUEUE_FILE + ".tmp"
    with open(tmp, "w") as f:
        json.dump(q, f)
        f.flush()
        os.fsync(f.fileno())
    os.replace(tmp, QUEUE_FILE)
except Exception:
    pass
```

Note the **atomic write-rename** pattern: a temporary file is flushed and synced, then atomically renamed. A process killed mid-write cannot corrupt the JSON; readers always see either the old or new state.

### Fallback path: legacy JSON→DB migration

`get_queue()` (config.py, lines 595–615):
```python
if not items and os.path.exists(QUEUE_FILE):
    try:
        with open(QUEUE_FILE, "r") as f:
            file_data = json.load(f)
    except Exception as e:
        file_data = []
        _log.warning("get_queue: could not read legacy %s: %s", QUEUE_FILE, e)
    # Per-row so one corrupt legacy item is skipped (and logged) rather
    # than silently aborting the entire migration.
    for item in file_data:
        try:
            queue_schema.migrate_legacy(item)
            q_item = QueueItem(**_split_queue_item(item))
            session.add(q_item)
        except Exception as e:
            _log.warning("get_queue: skipping un-migratable legacy item: %s", e)
```

On first run with a legacy `queue.json`, the DB is empty. `get_queue()` loads the JSON, applies `migrate_legacy()` to each item, and persists to the DB.

## Cross-process queue concurrency (config.py)

### The lost-update race

Multiple writers (dashboard endpoints, `run_fleet`, worker subprocesses) can race:

```
Process A: get_queue() → modify → save_queue()
Process B:  get_queue() → modify → save_queue()  # Clobbers A's changes
```

`save_queue()` deletes all existing items and reinserts (lines 625–627):

```python
session.exec(select(QueueItem)).all() # load for session
for item in session.exec(select(QueueItem)).all():
    session.delete(item)
```

Two unserialized writers doing get→modify→save concurrently lose each other's edits.

### Solution: cross-process lock (flock)

Commits 56d74be and 61094b0 introduce two mechanisms:

**queue_lock()** context manager (config.py, lines 537–551):
```python
@contextlib.contextmanager
def queue_lock():
    """Hold an OS-level advisory lock around a get_queue → save_queue
    round-trip. Cooperative across processes (host runner + container
    dashboard + worker subprocesses) since they all open the same
    .lock file. Non-recursive: nesting will deadlock — keep critical
    sections short and avoid calling out to other queue operations
    while holding."""
    os.makedirs(os.path.dirname(_QUEUE_LOCK_FILE), exist_ok=True)
    with open(_QUEUE_LOCK_FILE, "a+") as fd:
        fcntl.flock(fd, fcntl.LOCK_EX)
        try:
            yield
        finally:
            fcntl.flock(fd, fcntl.LOCK_UN)
```

Uses `fcntl.flock(LOCK_EX)` on a shared lock file (`_QUEUE_LOCK_FILE = QUEUE_FILE + ".lock"`, config.py line 534). Survives process restarts (file-backed).

**mutate_queue(mutator)** helper (config.py, lines 650–671):
```python
def mutate_queue(mutator):
    """Atomic read→modify→write of the queue under the cross-process lock.

    save_queue does a blind delete-all + reinsert of the whole list, so two
    processes doing get_queue→modify→save_queue concurrently (host run_fleet +
    container dashboard + worker subprocesses) lose each other's edits. This
    runs the full cycle inside queue_lock() (an flock shared via the same
    SLOPFINITY_STATE_DIR), serialising all writers.
    ...
    """
    with queue_lock():
        q = get_queue()
        result = mutator(q)
        q = result if isinstance(result, list) else q
        save_queue(q)
        return q
```

Wraps the full get→modify→save cycle under the lock. The mutator function must be side-effect-free (no network, GPU, or blocking calls).

### Event-loop blocking fix (commit 61094b0)

Commit 61094b0 identifies that `mutate_queue()` and `mutate_config()` perform flock + SQLite I/O synchronously on the FastAPI event loop, blocking other concurrent requests. **Solution**: wrap these calls in `await asyncio.to_thread()` in async handlers.

Example from routers/queue.py (line 204):
```python
# Before (blocks event loop):
cfg.mutate_queue(lambda q: ...)

# After:
await asyncio.to_thread(cfg.mutate_queue, lambda q: ...)
```

This offloads the lock + I/O to a thread pool, freeing the event loop. Used in routers/queue.py (lines 204, 264, 300, 339, 367, 394, 485, 506, 526, 566) and broadcaster.py (line 147–148, line 307).

### Configuration lock (config_lock)

A separate flock serializes `load_config()` and `save_config()` RMW cycles (config.py, lines 677–687). The two locks (`queue_lock` and `config_lock`) are independent; holding one does not block the other (verified in test_config_lock_is_exclusive, lines 85–98).

## Failure modes & edge cases

### 1. Extra column missing on existing DBs

**Failure mode**: An existing `slopfinity.db` created before commit 56d74be lacks the `extra` column. First write after upgrade would attempt `INSERT` with unknown column name.

**Fix**: `db.py:_migrate_sqlite_columns()` (lines 17–49) runs on every `init_db()` and uses `ALTER TABLE ADD COLUMN IF NOT EXISTS` (via exception suppression). Re-runs are safe.

### 2. Re-migration of already-migrated items

**Failure mode**: `migrate_legacy()` is called on every `get_queue()` read if items don't have `schema_version == 2`. Without the idempotence guard, a pending item would be re-stamped with a fresh id and reset to "needs" status.

**Fix**: Line 29 of `queue_schema.py` checks `if item.get("schema_version") == SCHEMA_VERSION and "stages" in item: return item` — idempotent.

### 3. id-less rows from run_fleet append

**Failure mode**: `run_fleet` appends rows to the queue without an id. The old code set `item_dict.setdefault("id", make_id())` but the `default_factory` on the SQLModel was still invoked if the id wasn't passed to the constructor, causing churn.

**Fix**: Move id stamping into `_split_queue_item()` (line 564, before model construction) so the id is always explicitly passed. Verified in test_save_queue_stamps_stable_id (test_config_extras.py, lines 12–24).

### 4. Extra dict double-wrapping

**Failure mode**: A caller reads a flattened item, modifies it, and saves it again. If they accidentally re-wrap fields into an `extra` dict, `_split_queue_item()` could miss them on the next save.

**Fix**: `_split_queue_item()` lines 570–572 defensively merge any pre-existing `extra` dict:
```python
prev = item_dict.get("extra")
if isinstance(prev, dict):
    extra = {**prev, **extra}
```

Newer top-level unknowns win (they supersede stale nested copies).

### 5. Lost updates across processes

**Failure mode**: Two processes call `get_queue()` → modify → `save_queue()` concurrently. The blind delete-all in `save_queue()` causes the second writer to clobber the first.

**Fix**: All write paths must use `mutate_queue(mutator)`, which serializes via `queue_lock()`. Pure reads (`get_queue()` alone, `/queue/paginated`) stay lock-free.

**Verification**: test_mutate_queue_no_lost_updates_across_processes (test_config_extras.py, lines 138–179) spawns 5 processes, each appending 20 items concurrently. All 100 items land (no clobbering).

### 6. Event-loop blocking on flock + SQLite I/O

**Failure mode**: `mutate_queue()` was called directly in async route handlers, blocking the event loop on lock acquisition and DB commits. Concurrent requests starved.

**Fix** (commit 61094b0): Wrap in `await asyncio.to_thread()` in async contexts (routers/queue.py, routers/chat.py, broadcaster.py).

### 7. migrate_to_db.py bypassing _split_queue_item

**Failure mode**: The legacy migration script was re-dropping extra fields by directly filtering with `k in model_fields` instead of routing through `_split_queue_item()`.

**Fix** (commit 61094b0): Route through `cfg._split_queue_item(item)` before SQLModel construction (scripts/migrate_to_db.py, line 51).

## Residuals & future work

### Phase 2: Workers read stages exclusively

The legacy `status` field is retained for backward compatibility with the existing `run_fleet` runner. Phase 2 (not yet shipped) will rewrite workers to read and advance only the `stages` dict, entirely decoupling from the legacy flat status.

### Phase 3: Delta updates instead of delete-all

Current `save_queue()` always deletes all items and reinserts (lines 625–627). Under high queue volume and concurrency, this is wasteful. A future version should track item additions/modifications and upsert selectively.

### Atomic JSON backup

The JSON backup is synced *after* the DB commit and uses atomic rename-on-close. A process killed between DB commit and JSON write will have a stale queue.json. The DB will be the source of truth, but this is a known gap. Future work may sync atomically with the DB transaction or drop the JSON backup entirely if the DB becomes the sole source.

### CPU-mode resolution in scheduler (commit 61094b0)

The scheduler's `llm_cpu_mode` and `tts_cpu_mode` fields are now nested under the `scheduler` dict (config.py, line 182–183), not at the top level. The llm.py module correctly reads from `config["scheduler"]["llm_cpu_mode"]` (verified in test_llm_cpu_mode_reads_scheduler_bucket, test_config_extras.py, lines 73–82).
